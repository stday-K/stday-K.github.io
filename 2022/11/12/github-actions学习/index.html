CI/CD是什么

介绍

GitHub Actions 是一种持续集成和持续交付 (CI/CD) 平台，可用于自动执行生成、测试和部署管道。

  • 持续集成(CI) ：通过自动构建应用并运行不同级别的自动化测试（通常是单元测试和集成测试）来验证代码仓库的内容的修改；通俗来说，就是编写规则，检验内容修改

  • 持续交付(CD) ：自动将已验证的代码发布到存储库，并自动将应用发布到生产环境

一言蔽之，通过编写语句，来检验对代码仓库进行的修改和通过检验后，通过git给予的容器，利用actions语法，对代码进行操作；

Github允许用户把上述的操作写成独立文件(.yaml)，来运行集成命令；因此可以通过他人的action来完成需要的目的

GitHub 做了一个官方市场，可以搜索到他人提交的 actions。另外，还有一个 awesome actions 的仓库，也可以找到不少 action。

组成

  • Workflows（工作流程）

    工作流程是一个可配置的自动化过程，它将运行一个或多个作业。 工作流程由签入到存储库的 YAML 文件定义，并在存储库中的事件触发时运行，也可以手动触发，或按定义的时间表触发。

    工作流程在存储库的 .github/workflows 目录中定义，存储库可以有多个工作流程，每个工作流程都可以执行不同的任务集。 例如，您可以有一个工作流程来构建和测试拉取请求，另一个工作流程用于在每次创建发布时部署应用程序，还有一个工作流程在每次有人打开新议题时添加标签。

  • Events（事件）

    事件是存储库中触发工作流运行的特定活动。即设定工作流程触发的条件，如对某一部分分支或代码进行修改，则触发该事件。

  • Runners（运行程序）

    运行程序是触发工作流时运行工作流的服务器。每个运行程序可以一次运行一个作业。GitHub 提供 Ubuntu Linux、 Microsoft Windows 和 macOS 运行程序来运行您的工作流程; 每个工作流程运行在一个全新的、新配置的虚拟机中执行。

  • Jobs（工作）

    作业是工作流程在同一运行程序上执行的一组步骤，步骤按顺序执行，并相互依赖；由于每个步骤都在同一个运行程序上执行，因此可以将数据从一个步骤共享到另一个步骤。例如，您可以有一个生成应用程序的步骤，然后有一个测试生成的应用程序的步骤。

  • step（步骤）

    每个 job 由多个 step 构成，一步步完成

  • action （动作）

    Action 是 GitHub Actions 平台的自定义应用程序，它执行复杂但经常重复的任务；

    每个 step 可以依次执行一个或多个命令（action）。

    您可以编写自己的操作，也可以在 GitHub 市场的工作流中找到要使用的操作。

语法规则

GitHub Actions 的配置文件叫做 workflow （工作流程）文件，采用 YAML文件格式

工作流程在存储库的 .github/workflows 目录中定义，存储库可以有多个工作流程，每个工作流程都可以执行不同的任务集。

workflow 文件的配置字段非常多，详见官方文档。下面是一些基本字段。

  • name

    name字段是 workflow 的名称。如果省略该字段，默认为当前 workflow 的文件名。

  • on

    on 定义哪些事件可以触发工作流运行。

    • 指定事件触发

    • 使用筛选器

      某些事件具有筛选器，可让你更好地控制工作流的运行时间。

      例如，push 事件具有 branches 筛选器，该筛选器仅在发生目标为与 branches 筛选器匹配的分支的推送时（而不是在发生任何推送时）运行工作流。

  • jobs

    • 使用 jobs.<job_id>.name 设置作业名称，该名称显示在 GitHub UI 中

    • 使用 jobs.<job_id>.needs 标识运行此作业之前必须成功完成的所有作业。

    • 使用 jobs.<job_id>.runs-on 定义要运行作业的计算机类型

      如果使用 GitHub 托管的运行器，每个作业将在 runs-on 指定的运行器映像的新实例中运行。
